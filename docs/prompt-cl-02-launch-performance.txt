# 2. 目的
初回起動時の重さを減らし、ModelContainer タイムアウト → 一時ストアフォールバックの発生率を下げる。

# 3. 変更対象ファイル

# 4. 現状理解（コードから読み取った事実）

## MemoraApp.swift の現状
- `MemoraApp` の `.task` で `DebugLogger.shared.markAppStart()` → `loadModelContainer()` を呼ぶ
- `loadModelContainer` は `Task.detached` で ModelContainer を生成、5秒タイムアウト付き
- タイムアウト時は in-memory store にフォールバック
- `DebugLogger` は `markAppStart / markModelContainerReady / markAppReady` を持つ

## ContentView.swift の現状
- **`@StateObject private var bluetoothService = BluetoothAudioService()`** ← 起動直後に初期化
- **`@StateObject private var omiAdapter = OmiAdapter()`** ← 起動直後に初期化
- `.onAppear` で `configureOmiAdapterIfNeeded()` を呼ぶ

## 起動パフォーマンスの問題点
1. `BluetoothAudioService` が ContentView の init 時に即座に初期化される（BLE scan 開始の可能性）
2. `OmiAdapter` も同様に即座に初期化される
3. これらが ModelContainer 初期化と同時に走ると、5秒タイムアウトに押し込められる原因になる
4. DebugLogger に各ステップの所要時間ログが不足

# 5. 実装要件
1. `ContentView` の `BluetoothAudioService` と `OmiAdapter` を lazy または on-demand 初期化に変更
   - `@StateObject` は init 時評価が必須なので、`@LazyStateObject` パターンまたは `.onAppear` での遅延初期化を検討
   - または `@StateObject` を残しつつ、サービス内部の重い処理（BLE scan 等）を最初のユーザーアクションまで遅延
2. `MemoraApp` の `.task` にステップごとのタイミングログを追加
   - `DebugLogger` に `markStep(_ step: String)` 相当を追加するか、既存 `addLog` で `[LaunchTiming]` カテゴリを使う
3. home 起動直後に不要な device scan / adapter init / heavy index rebuild が走らないようにする

# 6. 実装方針（推奨アプローチ）
1. `DebugLogger` に `markLaunchStep(_ label: String)` を追加（既存 addLog のラッパーでよい）
2. `MemoraApp.loadModelContainer` の各段階に `markLaunchStep` を仕込む
3. `ContentView` では `BluetoothAudioService` と `OmiAdapter` の初期化を `.task` または `.onAppear` の非同期ブロック内に移す
   - SwiftUI の `@StateObject` 制約上、プロパティラッパーは変更できない場合、サービス内部の init を軽くして scan 等を別メソッドに分離し、`.onAppear` から呼ぶ形でも可
4. 結果として main-thread の起動直後 workload を減らす

# 7. 受け入れ条件
- 起動直後の main-thread work が減る（BluetoothAudioService / OmiAdapter の重い初期化が遅延される）
- デバイス連携系が home 起動時に必ず初期化されない
- どこが遅いか追えるログが残る（DebugLogger に起動タイミングが記録される）
- ModelContainer の5秒タイムアウトが発生しにくくなる
