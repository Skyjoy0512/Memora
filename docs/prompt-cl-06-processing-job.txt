# 4. 現状理解（コードから読み取った事実）

## ProcessingJob.swift の現状
```swift
@Model
public final class ProcessingJob {
    public var id: UUID
    var audioFileID: UUID
    var jobType: String        // "transcription" | "summary" 等
    var status: String         // "pending" | etc.
    var progress: Double = 0
    var error: String?
    var startedAt: Date?
    var completedAt: Date?
}
```
- **stage 情報がない**（transcribing / summarizing / extractingMetadata 等の現在段階が不明）
- **retryCount がない**
- **maxRetries がない**

## PipelineCoordinator.swift の現状
- `PipelineEvent` enum で `.stepStarted / .stepCompleted / .chunkProgress / .completed / .failed` を通知
- `runTranscriptionPipeline / runSummaryPipeline / runFullPipeline` の3パスがある
- **ProcessingJob を全く使っていない** — event stream だけで進捗を通知、DB への永続化なし
- 失敗時は `.failed(step:error:)` event だけ発火し、再試行の仕組みがない

## FileDetailViewModel.swift の現状
- `handleTranscriptionPipelineEvent / handleSummarizationPipelineEvent` で event を受けて UI 更新
- 失敗時は `errorMessage` を表示するだけ
- **再試行ボタンがない**（ユーザーは手動で再度文字起こしボタンを押す必要がある）

# 5. 実装要件

### ProcessingJob モデルの拡張
1. `stage: String` を追加 — 現在のパイプライン段階（transcribing / summarizing / extractingMetadata / extractingTodos / finalizing）
2. `retryCount: Int` を追加 — デフォルト 0
3. `maxRetries: Int` を追加 — デフォルト 1
4. `createdAt: Date` を追加
5. ステータス更新ヘルパー:
   - `markStarted(stage:)` — status = "running", startedAt = now, stage 更新
   - `updateProgress(_ value: Double, stage: String)` — progress / stage 更新
   - `markCompleted()` — status = "completed", completedAt = now, progress = 1.0
   - `markFailed(_ error: String, stage: String)` — status = "failed", error 保存
   - `canRetry: Bool` — retryCount < maxRetries

### PipelineCoordinator への統合
1. `executeTranscription` の開始時に ProcessingJob を作成して SwiftData に保存
2. 各 PipelineEvent yield の前に ProcessingJob を更新:
   - `.stepStarted(.transcribing)` → `job.markStarted(stage: "transcribing")`
   - `.stepCompleted(.transcribing)` → `job.updateProgress(0.6, stage: "transcribing")`
   - `.stepStarted(.finalizing)` → `job.updateProgress(0.9, stage: "finalizing")`
   - `.completed` → `job.markCompleted()`
   - `.failed` → `job.markFailed(error.localizedDescription, stage: ...)`
3. `runFullPipeline / runSummaryPipeline` でも同様に ProcessingJob を更新

### FileDetailViewModel の再試行準備
1. 失敗時の event handler で ProcessingJob を記録
2. `retryLastFailedJob()` メソッドを追加:
   - ProcessingJob の retryCount をインクリメント
   - 元のパイプラインを再実行
3. UI 側での再試行ボタンはこのタスクでは必須ではないが、ViewModel に導線を用意する
