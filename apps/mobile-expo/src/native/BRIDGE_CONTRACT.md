# MemoraNative Bridge Contract

This document defines the native bridge surface between the Expo app and native-owned data.

Read before native work:

- `../../../docs/react-native-expo-migration-plan.md`
- `../../../docs/transcription-core-boundary.md`
- Expo Modules API docs: https://docs.expo.dev/modules/overview/

## Current State

- `MemoraNative.ts` is the React Native facade consumed by screens and hooks.
- Screens already consume the facade through hooks.
- A local Swift Expo Module shell exists at `modules/memora-native`.
- iOS native folders are generated and `MemoraNative` is detected by Expo autolinking.
- The Swift module uses app-sandbox audio files and JSON metadata for native-file list/detail. It is not connected to the existing Memora SwiftData/repository or STT services.
- The RN facade prefers the native module on non-web platforms.
- Web explicitly uses the mock/fallback path so UI review remains available without a native binary.
- Native audio-file list/detail calls are routed through public `MemoraAudioFileReading`; the current registry implementation is `MemoraNativeFileAudioFileStore`.
- Native audio-file create/rename/move/delete calls are routed through public `MemoraAudioFileMutating`; the current registry implementation is `MemoraNativeFileAudioFileStore`.
- Native settings read/write calls are routed through public `MemoraSettingsReadingWriting`; the current registry implementation is `MemoraUserDefaultsSettingsStore`.
- Native recording/import calls are routed through public `MemoraRecordingImportHandling`; the current registry implementation is `MemoraNativeFileRecordingImportHandler`.
- Native Ask AI query calls are routed through public `MemoraKnowledgeQuerying`; the current registry implementation is `MemoraSampleKnowledgeQuery`.
- Native-file metadata rename/move/delete calls are implemented for local JSON metadata generated by recording/import. Delete also removes the stored local audio file when present.
- Home currently exposes delete only for safe bridge-generated/native-file records.
- File Detail exposes inline title editing only for safe bridge-generated/native-file records and calls `renameAudioFile`.
- Settings calls `getBridgeInfo()` to show whether the app is using mock/sample/real data.

## DTO Rules

- DTOs must be JSON-friendly.
- Do not expose SwiftData model instances directly.
- Keep Swift-only types inside native code.
- Convert dates to ISO strings or app-facing display strings consistently.
- Do not modify STT core protected files just to satisfy RN DTOs.

## Read-only Audio File Adapter

Current Swift boundary:

```swift
public protocol MemoraAudioFileReading {
  var sourceDescription: String { get }

  func listAudioFiles() throws -> [MemoraAudioFileDTO]
  func getAudioFile(id: String) throws -> MemoraAudioFileDTO?
}

public enum MemoraNativeAudioFileReaderRegistry {
  public static var audioFileReader: MemoraAudioFileReading = MemoraNativeFileAudioFileStore()
}

public protocol MemoraAudioFileMutating {
  var sourceDescription: String { get }

  func upsertAudioFile(_ dto: MemoraAudioFileDTO, fileURL: URL) throws
  func renameAudioFile(id: String, title: String) throws -> MemoraAudioFileDTO?
  func moveAudioFile(id: String, projectId: String?) throws -> MemoraAudioFileDTO?
  func deleteAudioFile(id: String) throws -> Bool
}

public enum MemoraNativeAudioFileMutationRegistry {
  public static var audioFileMutator: MemoraAudioFileMutating = MemoraNativeFileAudioFileStore()
}
```

The current default reader loads JSON metadata produced by native-file recording/import and falls back to `MemoraSampleAudioFileReader` only when no local native-file metadata exists.
Its `sourceDescription` is `native-files`.
The current default mutator writes the same local JSON metadata and stored files.
Its `sourceDescription` is also `native-files`.

The first full real-data pass should define an adapter in the host app target that can see existing Memora SwiftData/repository types, then assign it to both `MemoraNativeAudioFileReaderRegistry.audioFileReader` and `MemoraNativeAudioFileMutationRegistry.audioFileMutator` during app startup.
This keeps the Expo module decoupled from app-target SwiftData model definitions while still giving React Native real read and mutation paths.
Current RN host-app injection point: `ios/MemoraRN/AppDelegate.swift` calls `MemoraNativeBridgeBootstrap.configureDefaults()` before React Native starts.
The implementation lives in `ios/MemoraRN/MemoraNativeBridgeBootstrap.swift`.
For real app data, keep `AppDelegate` thin and call `MemoraNativeBridgeBootstrap.configure(...)` from a host-app bootstrap path with concrete reader, mutator, recording/import, and settings adapters.

Example shape for the host app target, not yet implemented:

```swift
import MemoraNative
import SwiftData

struct SwiftDataAudioFileReader: MemoraAudioFileReading {
  let sourceDescription = "swiftdata"
  let repository: AudioFileRepositoryProtocol

  func listAudioFiles() throws -> [MemoraAudioFileDTO] {
    try repository.fetchPage(offset: 0, limit: 50).map(mapAudioFile)
  }

  func getAudioFile(id: String) throws -> MemoraAudioFileDTO? {
    guard let uuid = UUID(uuidString: id) else { return nil }
    return try repository.fetch(id: uuid).map(mapAudioFile)
  }
}
```

For mutations, use a separate or combined host-app adapter:

```swift
struct SwiftDataAudioFileMutator: MemoraAudioFileMutating {
  let sourceDescription = "swiftdata"
  let repository: AudioFileRepositoryProtocol

  func upsertAudioFile(_ dto: MemoraAudioFileDTO, fileURL: URL) throws {
    let file = AudioFile(title: dto.title, audioURL: fileURL.path)
    try repository.save(file)
  }

  func renameAudioFile(id: String, title: String) throws -> MemoraAudioFileDTO? {
    guard let uuid = UUID(uuidString: id), let file = try repository.fetch(id: uuid) else {
      return nil
    }

    file.title = title.trimmingCharacters(in: .whitespacesAndNewlines)
    try repository.save(file)
    return mapAudioFile(file)
  }

  func moveAudioFile(id: String, projectId: String?) throws -> MemoraAudioFileDTO? {
    guard let uuid = UUID(uuidString: id), let file = try repository.fetch(id: uuid) else {
      return nil
    }

    let targetProjectId = projectId.flatMap(UUID.init(uuidString:))
    try repository.move(file, toProjectID: targetProjectId)
    return mapAudioFile(file)
  }

  func deleteAudioFile(id: String) throws -> Bool {
    guard let uuid = UUID(uuidString: id), let file = try repository.fetch(id: uuid) else {
      return false
    }

    try repository.delete(file)
    return true
  }
}
```

Keep this adapter outside protected STT core files. If the existing repository needs app lifecycle or SwiftData container injection, add that injection at the Expo module/app startup boundary instead of modifying STT services.
When editing `AppDelegate.swift`, import the module as `internal import MemoraNative` to match Expo's generated provider import access.
For new RN host Swift files that import `MemoraNative`, use `internal import MemoraNative` for the same reason.
`getBridgeInfo().isRealDataConnected` becomes `true` only when both the audio-file reader and mutator report `sourceDescription == "swiftdata"`. This prevents a partially wired host target from being reported as a verified shared-data connection.

`getBridgeInfo().persistenceScope` reports where the current bridge data lives:

- `app-sandbox`: RN-local files in the RN app container
- `app-group`: an explicitly configured shared App Group container
- `shared-swiftdata`: a verified shared SwiftData store/context boundary
- `mock`: Web/mock fallback

The current RN default is `app-sandbox`. A package link alone does not share the existing SwiftUI app's `Application Support/Memora/Memora.store` because the RN app has a separate bundle/sandbox.

Important target-membership note from 2026-07-09 inspection:

- Existing `AudioFile`, `AudioFileRepositoryProtocol`, and `AudioFileRepository` live in the original app target under `Memora/Core`.
- The generated `MemoraRN` Xcode target currently compiles only RN host files and Expo/CocoaPods support files, not existing `Memora/Core` files.
- Before implementing `SwiftDataAudioFileReader` or `SwiftDataAudioFileMutator`, decide whether to share the required model/repository files into the RN target, extract a small shared Swift package/framework, or expose a host app service layer that owns `ModelContainer`.
- Do not make the Expo module itself import app-target internals. Keep the Expo module dependent only on DTO/protocol boundaries.

## Rename, Move, And Delete Rules

Current bridge methods:

```ts
renameAudioFile(id: string, title: string): Promise<AudioFileDTO | null>
moveAudioFile(id: string, projectId: string | null): Promise<AudioFileDTO | null>
deleteAudioFile(id: string): Promise<boolean>
```

Current local native-file behavior:

- `renameAudioFile` trims the title, rejects an empty title, updates JSON metadata, and returns the updated DTO.
- `moveAudioFile` accepts a project UUID or `null` for Inbox, persists the destination, and returns the updated DTO. The shared-store adapter rejects invalid project UUIDs before repository lookup.
- `deleteAudioFile` removes the JSON metadata entry and deletes the stored file when the metadata points at an existing local file.
- `deleteAudioFile` returns `false` for unknown IDs so RN can avoid pretending that sample/mock data was deleted.
- `startRecording`/`stopRecording` and `importAudio` persist returned DTOs through `MemoraNativeAudioFileMutationRegistry.audioFileMutator`.
- Web fallback only mutates generated in-memory review-session files.

UI rules:

1. Only expose destructive actions for records known to be generated by the bridge/native-file path.
2. Do not expose delete for sample/mock records.
3. Keep confirmation/error UI in RN, but keep actual file deletion in Swift/native code.
4. Only expose title editing for records known to be generated by the bridge/native-file path until a real repository adapter can authorize edits.
5. Keep the card open and delete controls as separate press targets; do not nest `Pressable` delete controls inside another `Pressable` on web.
6. When replacing local JSON metadata with SwiftData/repository persistence, route create/rename/move/delete through a host-app adapter instead of making the Expo module import app-target models directly.

## Processing Retry Queue

Failed transcription and summary requests can be persisted without coupling React Native to the protected STT or AI implementations.

```ts
type ProcessingRetryOperation = 'transcription' | 'summary'

enqueueProcessingRetry({
  audioFileId: string,
  operation: ProcessingRetryOperation,
  lastError?: string
}): Promise<ProcessingRetryDTO>

listProcessingRetries(): Promise<ProcessingRetryDTO[]>
recordProcessingRetryFailure(id: string, lastError: string): Promise<ProcessingRetryDTO | null>
completeProcessingRetry(id: string): Promise<boolean>
```

Current native behavior:

- Items are stored in `Documents/MemoraNativeMetadata/processing-retries.json`.
- The same `audioFileId` and operation are deduplicated; enqueue updates the latest error instead of creating an unbounded duplicate.
- `recordProcessingRetryFailure` increments `attemptCount` and persists the latest error.
- `completeProcessingRetry` removes a successfully handled item.
- Only `transcription` and `summary` are accepted. The queue does not call STT or AI services itself; a future host worker owns execution and calls the queue lifecycle methods.
- The file-backed queue accepts an injected storage URL for deterministic tests.
- Web review uses an in-memory fallback with the same contract.

`getBridgeInfo()` should report:

```ts
{
  platform: 'ios' | 'web' | string;
  moduleName: 'MemoraNative' | string;
  moduleVersion: string;
  audioFileSource: 'mock' | 'sample' | 'native-files' | 'swiftdata' | string;
  audioFileMutationSource: 'mock' | 'sample' | 'native-files' | 'swiftdata' | string;
  recordingSource: 'mock' | 'sample' | 'native' | 'native-file' | string;
  settingsSource: 'mock' | 'memory' | 'userdefaults' | 'keychain' | string;
  knowledgeQuerySource: 'mock' | 'sample' | 'native' | 'swiftdata' | string;
  summarySource: 'mock' | 'sample' | 'native' | 'swiftdata' | string;
  retryQueueSource: 'mock' | 'native-file' | string;
  persistenceScope: 'mock' | 'app-sandbox' | 'app-group' | 'shared-swiftdata' | string;
  isRealDataConnected: boolean;
}
```

## Recording And Import Rules

`startRecording`, `stopRecording`, and `importAudio` currently route through a public registry boundary:

```swift
public protocol MemoraRecordingImportHandling {
  var sourceDescription: String { get }

  func startRecording() throws -> MemoraRecordingSessionDTO
  func stopRecording(sessionId: String) throws -> MemoraAudioFileDTO
  func importAudio(uri: String) throws -> MemoraAudioFileDTO
}

public enum MemoraNativeRecordingImportRegistry {
  public static var handler: MemoraRecordingImportHandling = MemoraNativeFileRecordingImportHandler()
}
```

The default handler now performs native file recording/import inside the local Expo module:

- `startRecording()` requests microphone permission and records an `.m4a` file with AVFoundation.
- `stopRecording(sessionId:)` stops the matching recorder and returns a JSON-friendly `AudioFileDTO`.
- `importAudio(uri:)` copies the selected file URI into app Documents and returns a JSON-friendly `AudioFileDTO`.
- `sourceDescription` is `native-file`.

This creates local files, but it still does not persist records into the existing SwiftData/repository source of truth.
`MemoraSampleRecordingImportHandler` remains available as a safe fallback/reference handler.

When replacing the default file handler with full app integration:

1. Add adapter files at the host-app boundary.
2. Keep AVFoundation/file-write behavior inside Swift.
3. Return `AudioFileDTO` values to React Native rather than SwiftData models.
4. Do not modify protected STT core files as part of recording/import bridge work.
5. Assign `MemoraNativeRecordingImportRegistry.handler` during RN app startup after the adapter owns its `AudioRecorder`, import service, and SwiftData/repository dependencies.
6. Keep `NSMicrophoneUsageDescription` in both `app.json` and the generated iOS `Info.plist`.

Known existing boundaries inspected on 2026-07-09:

- `AudioRecorder` exposes start/stop recording behavior and returns native file URLs/results.
- `AudioFileImportService` imports a file into app storage and SwiftData using `ModelContext`.
- `CaptureSourceRegistry` provides capture/import source coordination through an `ImportSink`.
- `RecordingViewModel` already saves recordings through an audio-file repository protocol.

Keep the real adapter outside protected STT core files. If it needs `ModelContext`, repository, or app lifecycle ownership, pass those in from the host app target instead of making the Expo module depend directly on app-target models.

## Settings Rules

`SettingsDTO` currently contains non-secret UI state only:

```ts
{
  transcriptionMode: 'local' | 'api';
  summaryProvider: 'OpenAI' | 'Gemini' | 'DeepSeek' | 'Local';
  speechAnalyzerEnabled: boolean;
}
```

The current native implementation persists non-secret settings with `UserDefaults` so RN can consume the facade across app restarts.
The web fallback persists the same non-secret DTO to `localStorage` for quick design review.
The full app adapter should read/write through the existing Swift settings/keychain surfaces where appropriate.
Do not send raw API keys or secrets into React Native state.

## Summary Generation Rules

`generateSummary` currently accepts a JSON-friendly request and returns the existing summary DTO shape:

```ts
type SummaryRequestDTO = {
  audioFileId: string;
  options: {
    provider: 'OpenAI' | 'Gemini' | 'DeepSeek' | 'Local';
    templateId?: string;
  };
};
```

The current Swift boundary is `MemoraSummaryGenerating`, registered through `MemoraNativeSummaryRegistry.summaryGenerator`.
When the RN host opens the shared SwiftData store, `MemoraSharedStoreSummaryGenerator` is installed and reports `summarySource: 'swiftdata'`. It obtains the transcript and `AudioFile`, reads the selected provider key only from the RN Keychain service, constructs the provider in the host target, invokes `MemoraSharedSummary`, then persists `summary`, `keyPoints`, `actionItems`, and `isSummarized`.

The DTO, events, logs, and errors must never include an API key. Missing keys and unavailable shared storage return explicit safe errors; they must not fall back to sample output. Do not call provider clients or Keychain from the local Expo module.
