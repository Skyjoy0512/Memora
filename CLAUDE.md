# Memora - Claude Code 協働ルール

## 基本ルール

### コミュニケーション
- **すべての回答は日本語**で行う
- 専門用語はできるだけわかりやすく説明する
- 実装前に、何をするかを日本語で短く整理する

### 作業の進め方
大きな変更の前は以下の順で進める：
1. 現在のコードやファイル構成を確認
2. 現状理解を日本語で整理
3. 実装計画を短く示す
4. その後に編集

### 出力時のルール
- 変更するファイル名を先に示す
- 変更理由を簡潔に説明する
- モック実装なら明記する
- いきなり大規模変更をしない

## 開発方針

### 1. シンプルで保守しやすい構成
- 過度な抽象化を避ける
- まずは動くものを作る
- 将来の拡張性より現在のわかりやすさを優先

### 2. iOSアプリとしての品質
- iOS ネイティブの操作感を重視
- Human Interface Guidelines に準拠
- Apple の最新ベストプラクティスに従う

### 3. わかりやすさ優先
- コードはシンプルに
- コメントは必要最低限
- 複雑なロジックは分割

### 4. コストを意識した実装
- 無料サービス・低コストサービスを優先
- 必要な機能から段階的に実装
- 高価なサービスは代替案を検討

### 5. Windows / macOS の両方で継続しやすい構成
- IDE 依存を最小限に
- スクリプトやツールはクロスプラットフォーム対応
- 設定ファイルは OS 共通で扱えるように

## UI / UX 方針

### 基本スタイル
- **クリーンで上品**: ミニマルで洗練されたデザイン
- **モノトーン基調**: グレースケールをベースにした配色
- **読みやすさ最優先**: フォントサイズ、余白、コントラストに配慮

### 情報デザイン
- **情報の階層を明確に**: 重要な情報が目立つように
- **装飾過多にしない**: 必要な場面だけ視覚的アクセントを追加
- **iOSらしい滑らかさ**: アニメーション、トランジションを自然に

### 操作体験
- **直感的な操作**: 意図した操作ができるように
- **フィードバック**: 状態変化を適切に伝える
- **エラー対応**: わかりやすいエラーメッセージとリカバリー方法

## アプリ仕様

### 基本構造
- **ホーム画面 = Filesタブ**: 録音ファイル一覧を表示
- **要約フロー**: 文字起こし完了後に自動生成
- **話者分離**: 初期は Speaker 1 / Speaker 2 などの仮ラベルで実装

### UX 原則
- **エラー表示**: 軽いトースト表示で伝える
- **添付資料**: 保存とプレビューを優先
- **追加導線**: Project / File 追加がわかりやすい場所に配置

## 禁止事項

### セキュリティ
- **APIキーや秘密情報をコミットしない**
- `.env` ファイルを不用意に読んだり編集しない
- 有料サービスを勝手に前提にしない

### 変更管理
- 明示的依頼なしに大きく仕様変更しない
- ユーザーの確認なしに設定を変更しない
- コードの大規模リファクタリングは相談してから

## 作業完了時の報告

作業完了時は以下を日本語でまとめる：
- 何を変更したか
- 次にやるべきこと
- macOS / Xcode 側で必要な追加作業があるか

## 技術スタック

- **言語**: Swift
- **UI フレームワーク**: SwiftUI
- **プラットフォーム**: iOS (iOS 17+)
- **開発環境**: Xcode 15+ / VS Code (Windows)
- **アシスタント**: Claude Code
